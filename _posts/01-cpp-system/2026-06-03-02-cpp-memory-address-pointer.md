上一篇文章讲了变量、类型和值。

这篇进入 C++ 里最重要也最容易混乱的一层：

```text
内存、地址和指针
```

C++ 里的很多概念都绕不开内存。

读者一旦看见：

```cpp
T*
&x
*p
this
nullptr
```

就已经进入“对象在哪里、谁在访问它、它还活着吗”这一类问题。

## 1. 变量在内存里有位置

变量不是漂浮在代码里的名字。

例如：

```cpp
int x = 10;
```

可以先理解成：

```text
程序创建了一个 int 类型的变量 x。
x 在内存里占据一块空间。
这块空间里存着值 10。
```

变量名 `x` 是给这块内存起的名字。

后面写：

```cpp
x = 20;
```

不是重新发明一个 `x`。

而是把 `x` 对应那块内存里的值从 `10` 改成 `20`。

## 2. 地址是什么

每块内存都有位置。

这个位置就叫地址。

C++ 里可以用 `&` 取变量地址：

```cpp
int x = 10;
int* p = &x;
```

这里：

```text
&x 表示 x 的地址。
p 是一个指针变量。
p 保存的是 x 的地址。
```

注意：

```text
x 的类型是 int。
&x 的类型是 int*。
```

`int*` 读作：

```text
指向 int 的指针。
```

## 3. 指针是什么

指针是保存地址的变量。

```cpp
int x = 10;
int* p = &x;
```

可以画成：

```text
x:  [10]
      ^
      |
p ----+
```

`p` 自己也是一个变量。

它的值是一个地址。

这个地址指向 `x`。

所以指针声明要这样读：

```cpp
int* p;
```

意思是：

```text
p 是一个指针。
它指向 int 类型的对象。
```

如果是：

```cpp
Packet* p;
```

意思是：

```text
p 是一个指针。
它指向 Packet 类型的对象。
```

如果是：

```cpp
QbbNetDevice* dev;
```

意思是：

```text
dev 是一个指针。
它指向 QbbNetDevice 类型的对象。
```

## 4. 解引用是什么

指针保存地址。

如果要通过地址访问那个对象，要使用 `*`。

```cpp
int x = 10;
int* p = &x;

int y = *p;
```

这里：

```text
p  是地址。
*p 是 p 指向的那个 int 对象。
```

所以：

```cpp
int y = *p;
```

意思是：

```text
读取 p 指向的 int 值，把它放进 y。
```

也可以通过指针修改原对象：

```cpp
*p = 20;
```

这会把 `x` 改成 `20`。

因为 `p` 指向的就是 `x`。

## 5. nullptr 是什么

指针可以不指向任何对象。

C++ 里用：

```cpp
nullptr
```

表示空指针。

```cpp
int* p = nullptr;
```

意思是：

```text
p 是一个指向 int 的指针。
但它现在没有指向任何 int 对象。
```

空指针不能解引用。

下面这种代码是危险的：

```cpp
int* p = nullptr;
int x = *p;  // 错误：p 没有指向有效对象
```

读源码时，如果看到裸指针，就要问：

```text
这个指针有没有可能是 nullptr？
使用前有没有检查？
```

## 6. -> 是什么

如果指针指向一个对象，可以用 `->` 访问成员。

例如：

```cpp
Ptr<Packet> p = Create<Packet>(1000);
uint32_t size = p->GetSize();
```

这里的 `Ptr<Packet>` 是 ns-3 的智能指针。

它表现得很像指针，所以可以用：

```cpp
p->GetSize()
```

如果是裸指针，也类似：

```cpp
Packet* p = ...;
uint32_t size = p->GetSize();
```

`p->GetSize()` 大致等价于：

```cpp
(*p).GetSize()
```

只是 `->` 更方便。

## 7. this 是什么

在类的成员函数里，`this` 表示：

```text
当前对象的地址。
```

例如：

```cpp
class Foo {
public:
    void Bar() {
        // this 指向当前 Foo 对象
    }
};
```

如果 `Foo` 的某个对象调用：

```cpp
foo.Bar();
```

那么在 `Bar()` 里面：

```text
this 指向 foo 这个对象。
```

所以 `this` 是一个指针。

在 `QbbNetDevice` 成员函数里，`this` 的类型大致是：

```text
QbbNetDevice*
```

在 `RdmaHw` 成员函数里，`this` 的类型大致是：

```text
RdmaHw*
```

## 8. ns-3 事件系统里的 this

代码形态：

```cpp
Simulator::Schedule(
    txCompleteTime,
    &QbbNetDevice::TransmitComplete,
    this);
```

这行的意思是：

```text
txCompleteTime 之后，
在当前 QbbNetDevice 对象上调用 TransmitComplete。
```

这里的 `this` 是裸指针。

事件系统会保存它，等未来仿真时间到了再调用。

这会带来一个生命周期问题：

```text
如果事件触发前 this 指向的对象已经销毁，会发生什么？
```

答案是：

```text
可能访问悬空指针。
```

所以用 `this` 调度事件时，要保证对象活到事件执行，或者在销毁前取消事件。

## 9. 裸指针和 Ptr<T>

裸指针：

```cpp
Packet* p;
```

只保存地址。

它不自动管理对象生命周期。

ns-3 智能指针：

```cpp
Ptr<Packet> p;
```

不只是保存地址。

它还和 ns-3 的引用计数机制配合。

所以：

```text
T*      更像“保存对象地址”
Ptr<T>  更像“持有这个 ns-3 对象的一份引用”
```

两者都能通过 `->` 访问对象。

但生命周期含义不一样。

## 10. GetBuffer() 里的裸指针

代码来源：

```text
src/network/model/packet.h
```

```cpp
uint8_t* GetBuffer() const;
```

这个函数返回：

```text
uint8_t*
```

也就是：

```text
指向 uint8_t 的裸指针。
```

`uint8_t` 可以理解成一个字节。

所以 `uint8_t*` 经常表示：

```text
一段原始字节内存的起始地址。
```

这种指针很底层，也很危险。

如果代码通过偏移访问：

```cpp
uint8_t* buf = p->GetBuffer();
```

就要非常清楚：

```text
buf 指向哪里？
offset 是否正确？
这块内存还能不能访问？
```

## 11. 悬空指针

悬空指针指的是：

```text
指针还保存着某个地址，但那个地址上的对象已经不存在了。
```

中文里经常还会听到一个词：

```text
野指针。
```

这两个词有时候会被混着用，但更精确地说：

```text
未初始化指针：指针变量刚声明，没有被赋值，里面是什么地址不确定。
悬空指针：指针曾经指向一个有效对象，但这个对象后来已经不存在了。
野指针：口语里常泛指所有“不知道能不能安全访问”的危险指针。
```

所以：

```text
悬空指针通常也可以算作野指针的一种。
```

先看大括号 `{}`。

在 C++ 里，一对大括号可以形成一个代码块：

```cpp
{
    int x = 10;
}
```

这个代码块有自己的作用域。

在这个代码块里面声明的局部变量，出了这个代码块，生命周期就结束。

例如：

```cpp
int* p;

{
    int x = 10;
    p = &x;
}

// x 已经离开作用域
// p 还保存着 x 曾经的地址
```

这个例子要分三段看。

第一段：

```cpp
int* p;
```

此时 `p` 是一个未初始化指针，也常被叫野指针

它还没有被赋值，不应该读取它，更不能解引用它。

第二段：

```cpp
{
    int x = 10;
    p = &x;
}
```

在大括号里面，`x` 是一个有效的局部变量。

`p = &x;` 之后，`p` 保存的是 `x` 的地址。

在这个代码块还没结束时，`p` 指向的对象还活着。

第三段：

```cpp
// x 已经离开作用域
// p 还保存着 x 曾经的地址
```

离开大括号后，`x` 的生命周期结束。

此时 `p` 就是悬空指针。

也可以说：

```text
p 现在是一个危险的野指针。
```

但它不是“从一开始就乱指”的那种未初始化指针。

它的问题是：

```text
它记住了一个曾经正确、现在已经失效的地址。
```

如果再写：

```cpp
int y = *p;
```

行为就是未定义行为。

也就是说，程序可能看起来还能跑，也可能读到奇怪的值，也可能直接崩溃。

C++ 不保证结果。

读 ns-3 事件代码时，悬空指针尤其重要。

因为事件可能在未来才执行。

## 12. 读源码时的检查问题

看到指针相关代码，可以问：

```text
1. 这个变量是不是指针？
2. 它指向什么类型的对象？
3. 它有没有可能是 nullptr？
4. 谁保证它指向的对象还活着？
5. 使用的是裸指针 T*，还是 ns-3 的 Ptr<T>？
6. 如果传进事件系统，事件触发前对象会不会消失？
```

例如：

```cpp
Simulator::Schedule(delay, &Foo::Bar, this);
```

要读成：

```text
this 是当前 Foo 对象的地址。
事件系统会在未来使用这个地址调用 Bar。
因此 Foo 对象必须活到事件触发时。
```

## 13. 小结

这一篇的核心是：

```text
指针保存地址。
地址指向对象。
解引用通过地址访问对象。
this 是当前对象的地址。
nullptr 表示没有指向任何对象。
```

读 ns-3/RDMA 源码时，指针不只是语法。

它总是连着一个问题：

```text
对象还活着吗？
```
